1 Introduction

What is Security in Salesforce?

Salesforce security is a multi-layered framework designed to protect your data while still letting users collaborate easily. It's built on one guiding principle:

"Least Privilege — a user should be given the absolute minimum level of access necessary to do their job, and nothing more."

Think of Salesforce as a highly secure campus building that uses a series of checkpoints, each one narrower than the last:

🔐

Front Door

Org-Level — controls who can get into the building at all.

🚪

Rooms

Object-Level — determines which specific rooms a user can enter.

🗄️

Filing Cabinets

Field-Level — controls which cabinets inside those rooms can be opened.

📄

Files

Record-Level — dictates which specific files can be read or edited.

Why Do We Need Security?

🕵️

Data Protection & Privacy

Safeguard sensitive information from unauthorized access or breaches.

Example: a student's social security number can't be viewed by a standard lecturer.
🧩

Data Integrity

Prevent accidental — or intentional — data loss and corruption.

Example: if every student could edit the "Grades" table, the university's data would instantly lose its reliability.
⚖️

Regulatory Compliance

Organizations must follow strict data privacy laws — GDPR, HIPAA, FERPA for education.

Example: proper security settings keep the system legally compliant and audit-ready.

4 Organization-Level Security

Organization-Level Security (The Front Door)

Organization-level security controls the absolute outer perimeter of your org — dictating how, when, and from where a user can log in.

🛡️

MFA & SSO

Requires multiple proofs of identity, or a central company portal.

Example: students log in via campus email (SSO) and approve a phone prompt (MFA).
🌐

Network-Based Security (IP Ranges)

Blocks logins unless users connect from approved networks.

Example: external logins blocked unless from the university's internal network or VPN.
⌛

Time-Based Security (Login Hours)

Restricts access to specific days and times per profile.

Example: students can only log in from 6:00 AM to 10:00 PM.
🔑

Password Policies

Enforces password strength, update frequency, lockout attempts.

Example: a mix of letters/numbers/symbols, changed every 90 days.
⚙️

Session Settings

Manages active login periods and inactivity timeouts.

Example: students are auto-logged out after 15 minutes of inactivity.

EXERCISE · Organization-Level

Password Policy

User story: "As a University Admin, I want students to have secure passwords with complexity requirements and a 90-day expiration, so their accounts are protected from unauthorized access."

  • Navigate to Setup → Profiles → Student → Password Policies → Edit
  • Password Complexity: require letters, numbers & special characters
  • Password Expiration: 90 days
  • Minimum Password Length: 8 characters
  • Enforce Password History: 3 passwords remembered
5 Object-Level Security

Object-Level Security (The Rooms)

Profiles and permission sets grant CRUD (Create, Read, Edit & Delete) permissions on Salesforce objects.

👤

Profile

Defines how a user accesses objects and data. Assigned once per user.

Example: a Student profile grants Read on Course records, blocks Staff Payroll entirely.
➕

Permission Set

Extends a user's access without changing their profile.

Example: a Lecturer helping admissions gets an "Admissions" permission set, temporarily.
📦

Permission Set Group

Bundles multiple permission sets, usually by job function.

Example: one "Senior Lecturer" group instead of assigning three separate sets.

EXERCISE · Object-Level · Profiles

Lock down the Student profile

User story: "As a University Admin, I want students to view their grades, enrollments and course offerings, but completely prevent them from creating, updating, or deleting grades to maintain academic integrity."

  • Setup → Profiles → Student → Object Settings
  • Read-only on Course Offering, Enrollment, and Grade
  • No Create / Update / Delete on Grade
  • Read-only on Account fields like Account Number, Email, Phone
6 2026 Update

Permission Sets Lead, Profiles Remain

🔔 Update (2026): Salesforce cancelled the mandatory end-of-life date for permissions in Profiles, after customer feedback — profiles are not being forced out.

What changed

• Profiles keep working — no forced migration deadline.
• Existing profile-based orgs aren't at risk of losing access overnight.
• Profiles simply won't receive new features or investment going forward.

What Salesforce still recommends

• Lead with Permission Sets; use Profiles mainly for baseline org access.
• Assign multiple granular Permission Sets instead of cloning Profiles.
• Only ~20% of orgs have made the shift so far — get ahead of it.
• Keeps you aligned with Least Privilege (Module 1).

Source: Salesforce Help, "Permissions in Profiles Retirement Cancelled" (2026).

7 Field-Level Security

Field-Level Security (The Filing Cabinets)

FLS controls access to specific fields inside an object. Even if a user is allowed into the "room" (the object), FLS locks specific "drawers" (fields) — making them invisible or read-only.

Example: a recruiter can see the "Employee" object's names and job titles, but FLS completely hides "Salary" and "Social Security Number".

EXERCISE · Field-Level

Grade Override — restricted edit access

User story: "As a University Admin, I want only specific, authorized lecturers to override grades for students, ensuring standard lecturers cannot modify this field."

  • Setup → Permission Sets → PS_Lecturer → Object Settings → Grade__c
  • Field Permissions → locate GradeOverride__c
  • Check the Edit Access box, then Save
8 Record-Level Security

Record-Level Security (The Files in the Drawers)

🏛️

Organization-Wide Defaults (OWD)

The default access level to records a user doesn't own: Private, Public Read Only, Public Read/Write, Controlled by Parent.

Example: Grade OWD = Private, so students only ever see their own grade.
🏢

Role Hierarchy

Opens access vertically (upward) — managers automatically see what their subordinates see.

Example: the Dean sits above Lecturers, so the Dean sees every Lecturer-owned record.
📋

Sharing Rules

Automatic exceptions to OWD for particular groups of users.

Example: "Active" course offerings auto-shared with the "All Students" group.
✋

Manual & Apex Sharing

A one-off, user-driven way to open access on a single record.

Example: a lecturer going on leave manually shares their course with a substitute.

OWD vs. Sharing Rules — opposite jobs

Organization-Wide DefaultsSharing Rules
What it doesSets the baseline access for every user, org-wideOpens exceptions to that baseline for specific groups
DirectionRestricts — most locked-down setting for the orgExtends — never restricts further than OWD
Applies toEvery record of that object, every user, by defaultOnly matching records, shared with specific groups/roles

The OwnerId field

🧑

User ownership

OwnerId starting with 005. If you own a record, you automatically have full access to it — regardless of OWD.

Example: Lecturer Tim creates "Fall 2026 Biology" — he owns it, full control.
📥

Queue ownership

OwnerId starting with 00G. Any member of the queue can pick up and resolve the record.

Example: a student's IT Helpdesk ticket goes to the "IT Support Queue".

EXERCISE · Record-Level

Set Grade OWD to Private

User story: "As a University Admin, I want to set the baseline org-wide access for the Grade object to Private, ensuring users only see grades they own or have been explicitly granted access to."

  • Setup → Sharing Settings → Organization-Wide Defaults → Edit
  • Locate Grade (Grade__c)
  • Set Default Internal Access to Private
  • Set Default External Access to Private → Save
9 Automation

User Access Policies

Automated rules that grant or revoke user access based on criteria — assigning permission sets, permission set groups, licenses, public groups or queues the moment conditions are met.

Up to 20 actions / policy Up to 200 active policies / org
Why use it? Easier to manage in orgs with many users; dynamically assigns permissions with minimal manual work — e.g. new faculty accounts get the right permission sets and groups automatically, every week.
